## 需求背景

### 业务背景

sidecar 容器的镜像需要有明确的来源管理机制。当前存在两种候选方案（A：复用蓝鲸插件构建流程；B：独立构建接口），需先确定方案后实现对应的镜像管理能力。

### 用户故事

作为 **内部 AI 平台团队**
我想要 **sidecar 容器的镜像可以通过平台机制生成和管理**
以便于 **在创建 AI Agent 应用时选择合适的 sidecar 镜像**

### 详细功能描述

#### [F-002-1] 方案选型（方案 A vs B）

**方案 A：复用蓝鲸插件构建流程**
- 用户提交源码 → 平台完整构建流程 → 产出镜像
- 优点：可复用、流程完整
- 缺点：实现复杂

**方案 B：独立构建接口**
- 调用方通过独立 API 提交构建请求 → 镜像信息存入 DB
- 优点：简单、解耦
- 缺点：不易复用

> ⚠️ 本子需求启动前需完成方案评审决策

#### [F-002-2] 镜像构建/注册能力

- **输入**：源码/构建请求
- **处理逻辑**：
  - 方案 A：复用插件构建 pipeline，产出镜像并推送到镜像仓库
  - 方案 B：接收构建请求，执行构建，将镜像地址存入 DB
- **输出**：可用的 sidecar 镜像地址

#### [F-002-3] 镜像信息持久化与查询

- **输入**：镜像构建/注册完成事件
- **处理逻辑**：
  1. 新增 DB 模型存储 sidecar 镜像信息（镜像地址、版本、构建时间、关联应用等）
  2. 提供查询接口，返回可用的 sidecar 镜像列表
- **输出**：持久化的镜像记录 + 查询 API
